# APG Baseline深度分析与竞争策略

> [!abstract]
> 本文基于 AAAI 2026 论文《LLMs Unleashed: Generating Protocol Code from RFC Specifications》(APG) 与当前 `thesis_refactor_mainline_execution_plan_v5` 的战略重构，明确 Kiro 在 APG 出现后的定位、竞争策略、系统架构、实验设计与论文叙事。

## 一、结论先行

> [!important]
> APG 的出现并没有否定 Kiro 路线，反而帮助 Kiro 完成了从“RFC2Code 可行性探索”到“面向工业级交换机协议软件场景的工程完善”的战略转向。

当前最稳的定位不是：

- “我也要证明 LLM 能从 RFC 生成代码”

而是：

- “APG 已经证明了 RFC2Code 的可行性；Kiro 进一步解决其在可审计性、确定性、可修正性、平台解耦和分层验证上的工程缺陷”

这意味着后续主线应当明确分成两个阶段：

1. **Stage 1: Specification Grounding**
   - `PDF/RFC -> RAG 提取 -> 多层 IR -> 平台无关代码骨架`
   - 目标是得到可追溯、可验证、可重生成的协议语义骨架
2. **Stage 2: Environment Realization**
   - `Stage 1 产物 + platform profile + prompt/环境约束 -> 薄集成层`
   - 目标是在 Linux raw socket、DPDK、嵌入式 RTOS 等目标环境中落地

## 二、APG 到底做到了什么

### 2.1 基本信息

| 项目 | 内容 |
|------|------|
| 论文 | LLMs Unleashed: Generating Protocol Code from RFC Specifications |
| 会议 | AAAI 2026 |
| 代码 | `https://github.com/Assassin187/APG` |
| 协议 | ICMP, IGMP, NTP, TCP |
| 核心管线 | RFC Analyst -> Implementation Guidebook -> Network Protocol Coder -> Code Reviewer |
| 目标 | 端到端 RFC2Code 可行性验证 |

### 2.2 APG 的真实技术路线

APG 不是“有强类型 IR 的协议编译器”，而是一个 **prompt-driven guidebook-to-code** 系统：

```text
RFC 文档
  -> RFC Analyst 提取 Implementation Guidebook
  -> LLM 按协议类别直接/模块化生成 C/C++ 代码
  -> Code Reviewer 做编译检查与自动修补
```

它的核心中间产物 `Implementation Guidebook` 只是一种较浅的结构化描述，主要包含：

- 功能模块名
- 模块描述
- 对应 RFC 章节
- 函数签名/接口说明

它不是：

- 字段级偏移/位宽/约束的结构 IR
- 可独立验证的行为 IR
- 可确定性 codegen 的正式输入

### 2.3 APG 的实验亮点

它真正有说服力的点有三个：

1. **它把 RFC2Code 这条赛道做成了正式发表的问题**
2. **它做了真实系统实验，而不是只停在抽取**
3. **它展示了 Linux 用户态 raw socket 场景下的互操作**

尤其在 stateless 协议上，它的结果很亮眼：

- ICMP / IGMP / NTP 编译和行为正确率较高
- 速度上对 Sage 有一定优势

### 2.4 APG 不是“完全无人工介入”

> [!warning]
> APG 在简单无状态协议上接近零人工，但在 TCP 这类复杂有状态协议上并不是零人工。

从论文描述看，TCP 互操作成立依赖这些额外处理：

- 从 RFC 图示中额外提取 teardown 序列再注入 prompt
- checksum 逻辑由人工实现为可复用库函数
- prompt 中显式补字节序转换约束
- 对 SEQ/ACK 逻辑做进一步 prompt 强化和最小人工 patch

所以 APG 更准确的表述应是：

- **stateless 上高度自动化**
- **stateful 上 prompt 强化 + 最小人工修补**

### 2.5 APG 的“互操作”是如何实现的

APG 并不是把生成代码并入 Linux 内核协议栈，而是：

```text
生成的用户态 TCP 实现（基于 raw socket）
    <-> Linux 原生内核 TCP client/server
```

实验环境是 Ubuntu 用户态，抓包方式是 `tcpdump`，验证内容是：

- three-way handshake
- four-way teardown
- data transfer

这说明它证明的是：

- “LLM 生成的用户态协议原型可以和内核栈对话”

而不是：

- “LLM 直接生成了工业级协议栈实现”

## 三、APG 的核心弱点，也是 Kiro 的机会窗口

### 3.1 “全自动”有水分

APG 的 prompt 中硬编码了不少领域知识和平台规则，例如：

- raw socket 使用方式
- 字节序处理要求
- 图示优先于歧义文本的规则
- TCP 细节性的行为约束

这意味着：

- APG 的自动化很大程度依赖手工 prompt 工程
- 换协议族、换运行平台，往往需要重新设计 prompt

### 3.2 没有正式 IR 层

这是 APG 最关键的结构性短板。

因为没有强类型 IR，所以它存在这些问题：

- Guidebook 错误会静默传递到代码生成
- 每次生成都仍依赖 LLM 再“理解一次”
- 很难做字段级 readiness 校验
- 很难做 deterministic re-codegen
- 很难做小范围修正

### 3.3 验证层次不够深

APG 的 Code Reviewer 主要还是：

- `g++ -fsyntax-only`
- 编译失败后的自动修补

它没有系统性地做：

- roundtrip
- 字节级 pack/unpack 对照
- IR readiness 检查
- trace verify

### 3.4 平台耦合严重

APG 的可运行代码基本绑在 Linux 用户态 raw socket 假设上：

- socket 逻辑、字节序、checksum、平台行为混在一起
- 平台适配不是一个独立层
- 换目标环境时很难复用前面的协议语义产物

### 3.5 在真实工业场景中不够“spec coding”

APG 更像：

- `prompt-guided protocol prototyping`

Kiro 更应该追求：

- `spec-grounded protocol engineering`

两者的差异在于：

- APG 更强调“能生成并跑起来”
- Kiro 更强调“规范语义如何稳定地沉淀成可追溯、可修正、可验证的代码资产”

## 四、Kiro 应该如何扬长避短

### 4.1 不和 APG 硬拼的战场

不建议把主目标定成：

- “全面超过 APG 的 TCP 互操作表现”
- “直接一把生成完整协议栈并与内核对话”

这既不符合当前仓库成熟度，也不符合专硕项目的最优性价比。

### 4.2 应该放大的优势

Kiro 最值得放大的不是“直接生成终态代码”，而是这五点：

1. **多层 IR**
   - `MessageIR / BehaviorIR-lite / StateContextIR`
2. **可追溯**
   - evidence、页码、字段来源
3. **可修正**
   - patch lane、改 IR 再 codegen
4. **可验证**
   - compile、roundtrip、pcap、trace 的分层验证
5. **平台解耦**
   - Stage 1 固定协议语义，Stage 2 处理环境集成

### 4.3 最佳竞争叙事

> [!tip]
> 最稳的竞争话术不是“我在所有指标上都打败 APG”，而是“我解决了 APG 没有解决的工程化问题”。

建议统一成下面这条叙事：

```text
APG 证明了 RFC2Code 的可行性，
Kiro 进一步把它推进为一个面向工业级协议软件场景的
多层 IR 驱动、可审计、可修正、可验证、平台解耦的工程系统。
```

## 五、V5 的核心架构：两阶段解耦

### 5.1 总体设计

```text
Stage 1（平台无关，确定性）
  协议 PDF/RFC
    -> RAG 提取
    -> 多层 IR
    -> 确定性 codegen
    -> msg_*.h/c, fsm_*.h/c, roundtrip tests

Stage 2（平台相关，环境驱动）
  Stage 1 产物
    + platform profile
    + prompt/环境规则
    -> 薄集成层 protocol_app.c / Makefile / runtime glue
```

### 5.2 与 APG 的本质区别

| 维度 | APG | Kiro V5 |
|------|-----|---------|
| 生成阶段 | 一阶段耦合 | 两阶段解耦 |
| 协议语义与平台规则 | 混合在 prompt 中 | Stage 1 / Stage 2 分离 |
| Stage 1 可移植性 | 低 | 高 |
| 改字段后的重生成 | 重跑整条链 | 改 IR 后重跑 Stage 1 |
| 换平台代价 | 重写 prompt | 只换 Stage 2 |

### 5.3 两阶段架构的价值

这套设计解决了 APG 最难回答的问题：

- 协议语义错了，还是 socket/平台集成错了？
- 换目标环境时，之前的协议理解成果是否还能复用？
- 修一个字段、一个状态、一个定时器时，是否必须整份代码重做？

Kiro 的答案是：

- 协议语义固定在 IR 和 Stage 1
- 环境适配收敛在 Stage 2
- 语义修正和平台修正可以解耦

## 六、V5 执行主线

### 6.1 协议角色

| 协议 | 角色 | 目的 |
|------|------|------|
| FC-LS | 第一主案例 | 与题目“交换机协议软件”完全贴合，突出领域差异化 |
| BFD | 稳定基线 | 保护主链，承载 Stage 2 与 pcap 验证 |
| TCP Header | 结构挑战样例 | 与 APG 直接形成结构层对标 |
| ICMP/ARP | 可选泛化 | 视时间做简单协议展示 |

### 6.2 Phase 摘要

| Phase | 目标 |
|------|------|
| Phase 0 | 基线冻结 + GT 标注 + pcap 收集 |
| Phase 1 | FC 结构主线打通 |
| Phase 2 | pcap 字节级互操作验证 |
| Phase 3 | Stage 2 Linux 用户态适配 |
| Phase 4 | BehaviorIR-lite + FSM skeleton 稳定化 |
| Phase 5 | StateContextIR 最小落地 + trace verify |
| Phase 6 | 全部实验执行，对标 APG |
| Phase 7 | 回归、泛化、论文收束 |

### 6.3 优先级

> [!important]
> 当前必须做的是把最能支撑论文主线的数据先做出来，而不是平均推进所有模块。

**P0**

- Phase 0
- Phase 1
- Phase 2
- Phase 3
- Phase 6 中的结构层实验、LLM 对比实验、IR 价值实验

**P1**

- Phase 4
- Phase 5
- 行为层实验
- Stage 2 互操作实验
- 性能开销实验

**P2**

- 简单协议泛化
- Directed 模式
- patch lane 最小版展示

## 七、验证策略：这是 Kiro 真正能超过 APG 的地方

### 7.1 分层验证

```text
Layer 0: IR readiness
Layer 1: compile + symbol checks
Layer 2: roundtrip verify
Layer 3: pcap 字节级验证
Layer 4: Stage 2 Linux 用户态互操作
Layer 5: trace verify（行为层）
```

### 7.2 与 APG 的验证对比

| 验证层 | APG | Kiro V5 |
|--------|-----|---------|
| IR readiness | ❌ | ✅ |
| 编译检查 | ✅ | ✅ |
| Roundtrip | ❌ | ✅ |
| pcap 字节级验证 | ❌ | ✅ |
| raw socket 互操作 | ✅ | ✅ |
| trace verify | ❌ | ✅ |

### 7.3 为什么 pcap 验证特别重要

`pcap -> unpack -> 字段对照 -> pack -> memcmp` 这条链能让你避免陷入“必须先生成完整协议栈”的泥潭。

它有四个优点：

- 比“能不能通信”更细，能看到字段级和字节级正确性
- 离线可复现，不强依赖复杂网络环境
- 容易和 Wireshark/TShark 对齐
- 很适合当前 `MessageIR + pack/unpack` 主线

## 八、实验设计应该怎样对标 APG

### 8.1 APG 应该作为什么 baseline

APG 适合做：

- **主文献 baseline**
- **主对标工作**

但不适合做你唯一的实验 baseline。

你的实验 baseline 最稳的组合应该是：

1. **APG**
   - 用于论文 related work 和方法论对标
2. **直接 LLM 生成**
   - 用于主实验 baseline
3. **自身 ablation**
   - 如无 IR、无 Directed、无 patch lane、无 Stage 2

### 8.2 六个实验的定位

| 实验 | 内容 | 作用 |
|------|------|------|
| 实验一 | 结构层评估 | 对标 APG 的 Packet Struct / CSR |
| 实验二 | 行为层骨架评估 | 展示 APG 没有系统解决的行为层 |
| 实验三 | vs 直接 LLM 生成 | 证明 IR 中间层价值 |
| 实验四 | IR 层价值专项 | 展示确定性、可修正性、溯源能力 |
| 实验五 | Linux 用户态互操作 | 在 APG 主场直接对标 |
| 实验六 | 性能与开销分析 | 补足系统工程数据 |

### 8.3 最容易“超过 APG”的指标

如果只看“谁更像一把生成 TCP”，你未必占优。

但如果看这些指标，你很有机会更有说服力：

- 字段正确率
- roundtrip 通过率
- pcap 字节级一致率
- 生成一致性
- 错误可定位性
- 修正代价
- 平台代码隔离度

## 九、论文与答辩叙事建议

### 9.1 论文叙事主线

```text
协议软件开发成本高
  -> APG 证明 RFC2Code 可行
  -> 但 APG 缺乏 IR、分层验证和平台解耦
  -> 本文提出多层 IR + 两阶段代码生成系统
  -> 在 FC/BFD/TCP 上验证结构层、行为层与平台适配能力
  -> 证明系统更适合工业级协议软件辅助开发
```

### 9.2 最适合写进论文的定位句

> APG 证明了 prompt-driven RFC2Code 的可行性，尤其在无状态协议上效果显著；本文进一步引入多层 IR 与两阶段代码生成架构，将协议代码生成过程从黑盒 prompt 输出推进为可追溯、可修正、可验证、平台解耦的工程化流水线。

### 9.3 答辩时不要强调的点

- 不要强调“我也是首个 RFC2Code 系统”
- 不要把自己逼到“全面超越 APG 所有结果”
- 不要把 TCP 完整互操作当成唯一核心目标

### 9.4 答辩时应该强调的点

- APG 已经把赛道合法性证明了
- 你做的是工程完善，不是重复造轮子
- 你解决的是工业落地最需要的问题：可审计、可修正、可验证、可移植

## 十、一句话版战略判断

> [!success]
> APG 是可行性论文，Kiro 应该成为工程化论文；APG 的强项是“能生成”，Kiro 的强项应该是“生成过程可被相信、可被修、可被验证、可迁移到真实协议软件环境”。
