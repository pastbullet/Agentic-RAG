# 论文思路记录

## Schema 质量改进阶段的方案演化

### 一、当前问题的重新认识

在完成一轮 `MERGE Phase 2`、`EXTRACT` 精度修复和 `CODEGEN` 显示优化之后，系统已经能够跑通从提取、合并、生成到验证的完整 pipeline，但真实 BFD 文档的运行结果表明，当前瓶颈仍然集中在 schema 质量本身，而不是代码生成层。

具体来说，当前系统已经解决了一部分明显错误，例如：

- `Echo Packet` 不再被错误提取为固定字段结构；
- `Simple Password` 中的 `Password` 字段已经被修正为可变长度；
- timer 能够较稳定地合并为单个对象；
- 生成代码的文件命名也比之前更加清晰。

但是，真实运行结果也暴露出更深层的问题：

- 同一类状态机仍然会被拆成多个对象，去重不充分；
- 同一类报文结构仍然会因为标题差异、字段命名差异而保留多个版本；
- 这些重复对象继续向下游传递，最终在 `VERIFY` 阶段表现为重复 `case`、重复测试函数名等结构性冲突。

因此，问题的根源并不是“阈值不够合适”这么简单，而是**对象表示的一致性不足**：同一语义对象在不同章节中以不同名字、不同措辞、不同缩写方式出现，导致当前基于表面字符串的相似度计算低估了它们的真实接近程度。

### 二、对后续优化方向的思考

一开始的直觉是继续加强规则，例如进一步调低阈值、增加更多 merge 条件，或者把更多相似 pair 强行合并。但经过分析后发现，这种路线虽然短期可能提升召回，但中长期会使系统逐渐演化成一个“补丁式规则集合”：每遇到一种新的标题写法、字段写法、事件表达方式，就要补一条规则。这样做不仅工程维护成本高，而且论文方法论也会显得零散。

另一种思路是直接把是否合并交给 LLM。但这同样存在明显风险：LLM 对“相关”和“可合并”的边界并不总是稳定，很容易把语义接近但结构上不应合并的对象错误归并。如果让 LLM 直接成为最终裁决者，系统将失去规则系统原本具备的边界可控性和可解释性。

因此，后续优化不应简单地在“纯规则”和“纯 LLM”之间二选一，而应该将两者的职责重新划分，形成一个更稳定、更可解释的人机协同流程。

### 三、拟采用的完整优化路线

我目前认为最合适的路线是：

**先提升对象表示的一致性，再用规则做高置信自动合并，对模糊样本引入“LLM 证据整理 + 人工轻量确认”的机制，最终形成一个不依赖人工通读原文的合并流程。**

这个路线可以分为三层。

#### 1. 表示层规范化

第一层不是直接改 merge 阈值，而是先解决“为什么同一对象看起来不像同一对象”的问题。

在状态机上，需要重点提升三个方面：

- **名称匹配支持包含关系**  
  不能只看普通 token Jaccard，而要考虑长标题与简称、章节标题与图标题之间的覆盖关系。例如一个名称基本包含另一个名称的大部分核心词时，不应因为长度不同而被低估。

- **状态转移事件表达的规范化**  
  同一个事件在不同章节可能用不同短语表达，例如 “timer expires” 与 “when the timer has expired”。因此，事件比较应先做短语归一化，尽量统一词形、去除停用词、保留核心动作和条件，再进入转移重合度计算。

- **状态集合继续作为强特征，但不单独决定是否合并**  
  因为多个相关状态机可能共享相似的核心状态集合，所以状态重叠度是强信号，但必须和名称、转移结构一起使用，不能单独成为自动合并依据。

在报文结构上，重点是字段规范化：

- 字段名需要先做 canonicalization，处理缩写、全称、括号说明、连接符差异等；
- 报文名称只作为辅助特征，而字段结构才应是主特征；
- 在无互斥关键词冲突的前提下，字段集合高度一致的对象应被视为强候选。

这一层的目标不是直接决定是否合并，而是让后面的相似度计算更真实，避免“表面写法不同、语义却相同”的对象被系统误判为不相似。

#### 2. 高置信规则自动合并

第二层仍然保留规则主导的自动合并流程，但原则变为：

**只有结构证据已经足够强的对象，系统才自动合并。**

也就是说：

- 明显相同的对象由规则系统直接合并；
- 明显不同的对象由规则系统直接排除；
- 只有中间那一小部分“差一点”的模糊样本，才进入下一层处理。

这样做的核心目的，是保证自动合并部分具有较高 precision，不因为追求更多召回而把 schema 结构搞乱。

在论文叙述上，这一层可以强调：系统不是依赖单一名称相似度，而是采用“结构证据优先”的高置信自动聚合策略。

#### 3. LLM 证据整理 + 人工轻量确认

第三层是我认为最关键、也最适合项目定位的一层。

对于那些规则无法稳定判断、但又明显值得进一步检查的 near-miss pair，不再要求人工回到 RFC 原文重新比对，而是引入一个 **evidence-centered HITL** 机制。

它的核心思想是：

- 规则系统先筛出疑似重复但置信度不足的 pair；
- LLM 不直接输出 “merge / not merge”，而是把这两个对象整理成一张结构化的“证据卡”；
- 人工只看证据卡进行轻量确认，而不是重新通读协议原文。

这张证据卡中，理想上应包括：

- 共同点：共同状态、共同字段、共同事件、共同 source pages 分布特征；
- 差异点：只在一边出现的状态、字段、事件；
- 命名关系：是否为简称/全称、是否存在包含关系、是否存在缩写与全称映射；
- 结构判断线索：差异更像补充细节，还是更像本质不同；
- 系统建议：例如建议合并、建议保持分离、建议进一步人工确认。

在这一机制中：

- **规则系统负责筛选候选；**
- **LLM 负责整理证据；**
- **人工负责最终拍板。**

这使人工从“重新阅读 RFC 寻找依据”转变为“审核系统已整理好的证据并确认结论”，大幅降低了人工成本，同时保留了人在边界样本上的判断可靠性。

### 四、为什么这条路线更适合本课题

从工程角度看，这条路线的好处是：

- 改动可以按层推进，不需要一次性推翻现有 pipeline；
- 规则系统仍然负责主流程，整体行为可控；
- LLM 不承担最终裁决职责，边界漂移风险更低；
- 人工只介入少量高价值样本，系统不会退化为传统人工标注流程。

从研究和论文角度看，这条路线也更完整：

- 它不仅解释了“为什么当前 schema 会重复”，还给出了解决这一问题的结构化方法；
- 它可以自然组织成“表示层改进 → 自动聚合 → 证据驱动的人机协同消歧”的方法链条；
- 它既体现自动化能力，也体现系统在高不确定样本上的可解释性与可控性。

相比单纯强调规则增强，或者单纯强调 LLM 判断，这条路线更符合本项目“自动提取，但不要求人工全文介入”的目标，也更适合作为论文中的方法学核心部分。

### 五、当前阶段的具体判断

基于目前的开发进度，我认为下一阶段不应优先继续做模板层或代码生成层的包装，而应继续聚焦 schema 质量本身。

更具体地说，接下来的工作重点应是：

1. 进一步增强名称、事件、字段的规范化能力；
2. 让自动合并真正建立在结构证据之上；
3. 为未合并的 near-miss 对象建立诊断输出；
4. 在此基础上，再引入 evidence-centered 的 HITL 机制。

只有当 schema 层足够稳定、重复足够少时，下游的 `CODEGEN` 与 `VERIFY` 才会自然变得更可靠。
